VibeCoding 新手技巧：让 AI 先计划再工作

Make a plan before coding.

这是我在 Vibe Coding 初期，尝试下来觉得最有用的经验。。

不过，我依然踩了很多坑。直到最近我写完这篇文章，我才算真的理解。

（一）初次理解

当时我用的是 vscode 里的 Github Copilot，所以在 Youtube 上找了一个 Best Practice 的视频看。

印象最深的，就是视频里的工程师给了几句 Prompt，然后 copilot 像模像样地先列了一些 todo 再开始干。

虽然最后也是一步完成的，但是干的过程中，它会按照计划，一步完成，勾选，再开始下一步，直到结束。

可是那个视频本身也是很早以前的视频，讲演者并没有细致地演示她是怎么做的。

印象中，她也是和我一样，和 AI 聊天提需求。于是，我就这样稀里糊涂地开干了。

一开始，我就是在和 Copilot 干聊提需求。有的时候，它会先计划（列 TODO）再生成代码，但有的时候，它又不会这样，很不稳定。

（二）改进

我直觉上觉得：列计划还是比较好的。因为就算第一步走偏了，第二步看到原来的计划，也还能走回来。

所以，我改进了一下。每次都在 prompt 里和 AI 说：“Make a plan before coding”。

这样，似乎每次都比较稳定了。

可是，我又发现了一个新问题。有的时候，我给的指令本身就思考得不太全。

它是像模像样地列了个计划，但是目标本身有问题。 AI 一顿猛干，还是效果不好。

（三）再次改进

于是，我开始在 Ask 模式下，先和 AI 讨论计划，然后让它存成一个 Plan.md。然后，我再让 AI 参考这个 Plan 一步步操作。一开始这个方法还可以，AI 也是一步步操作。

但是，把计划存下来以后，我就会忍不住想看，想加东西。于是，这个 Plan 往往很大。

我开始不放心，我开始想让自己一步步监督 AI 实现。我改成在一个聊天或者多个聊天窗口里分别说：

Read the codebase carefully, and Implement only the Step1 of Plan.md

或者

Implement only Step2

人工操作多了以后，事情开始变得复杂了：

- AI 开发过程中发现出错了；
- AI 在修复过程中新增了额外的不在 Plan.md 的内容；
- 我在过程中看到 AI 的实现发现不符合我的要求，我顺口提出了需求变更；

……

我开始想办法让 AI 在开发的过程中，如果发现代码和文档不一致，更新文档。

但是，这么一来，手动要操作的事情越来越多。

比如不是每次 AI 开发完都后自己跑测试，即便我在 AGENT.md (copilot-instructions.md）里写了要求……

（四）

最近，Copilot 出了 /Plan 模式，不过我也没怎么研究。

因为，我开始使用 Claude Code 了。

这一次，我没有再着急做项目（因为急也没用）。

这两天看了 Claude Code 创建者的 Twitter 和它去年 4 月份发的 Claude Code 最佳实践。

我再一次看到这句：让 AI 先计划再开发。

Claude Code 的 /Plan 模式使用快捷键 Shift+tab+tab 激活。

和之前不一样的是，在我和 AI 聊完以后，它会直接问我是否接受这个计划，如果接受，它就会自动转成 Agent/Edit 模式开始实现。

有一次我肌肉记忆，还尝试过让它把 Plan.md 保存在本地，然后手动让它一个个实现。

随后，我再次遇到之前的各种问题。不过，我还是忍着干完了。

有一次，我突然发现 Claude 在 Plan 模式下，其实会在 ～/.claude 目录下保存一个 markdown 文档，而且还会随着我的需求持续更新，完全不需要我指定。

因为 Plan 模式到实际开发模式的切换过于顺畅，我几乎就是让 AI 自己干所有活了。

而且，这一切都发生在一个 Claude Code session 里，先计划再生成代码。

这么干了 1-2 天以后，我突然意识到，这才是真的 “Make a plan before coding”。

好的工具，会改变你的认知，真的。

所以，回到最开始的问题：

先计划再实现，其实就是在一次聊天（session） 里，先用 plan 模式和 AI 聊清楚需求，再切换成 Agent 模式，让 AI 直接开始实现。

在 Claude Code 里，甚至不用你手动切换模式，就这么简单。
