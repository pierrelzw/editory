# 本地 Codex Review vs 远程 Codex Review，到底该用哪个？

> 本地 Review 是你的随身镜子，远程 Review 是团队的安检门。两个都要。

---

前几天在配置 Claude Code 的 subagent 时，我顺手把 Codex CLI 也加进了 code review 流程——本地跑一个 `codex-code-reviewer`，写完代码直接审。

跑了几天之后，有朋友问我：GitHub 上不是也能配 Codex 做 review 吗？你本地再跑一个，不是重复了？

还真不是。用下来我觉得这两个东西解决的是完全不同的问题。

## 本地 Review：写代码时的即时反馈

本地 Codex Review 跑在你自己的机器上，通过 CLI 触发。

它的核心价值是**快**——改完代码，随手跑一下，几秒钟就能拿到反馈。不用 commit，不用 push，不用等 CI 排队。

我现在的习惯是写完一段逻辑就让它审一遍，有点像写作时边写边校对。很多低级错误——变量名拼错、边界条件漏了、类型不对——在这个阶段就被拦住了。

还有一个好处：它能看到你本地所有的代码，包括没提交的、没 push 的。有时候你改了三个文件但只 stage 了两个，远程 Review 根本不知道第三个文件的存在。

## 远程 Review：合并前的质量门禁

远程 Codex Review 跑在 CI 环境里（比如 GitHub Actions），通常在你创建或更新 PR 时自动触发。

它的核心价值是**强制性**和**可见性**——每个 PR 都必须过审，结果所有人都能看到。

审查意见直接以 PR comment 的形式出现，团队成员可以跟进讨论。你还可以把它设成 required check，不通过就不能合并。

但它也有局限：只能看到 PR 的 diff，看不到你本地还没 push 的改动；而且需要排队，反馈不是即时的。

## 两者是互补的

打个比方：

本地 Review 像你出门前照的那面镜子——快速检查，扣子有没有系错，领子有没有翻好。这是给自己看的，随时可以照。

远程 Review 像机场的安检门——标准化、强制、所有人都要过。它不关心你在家照没照镜子，它有自己的检查清单。

我现在的工作流是这样的：

1. **写代码时** — 本地 Review，即时反馈，快速迭代
2. **准备提 PR 时** — 本地再 Review 一次，确保没有明显问题
3. **PR 创建后** — 远程 Review 自动跑，作为团队的质量底线

两个都用，才是完整的。

你觉得哪个对你更有用？或者你有别的 Review 工作流？
