# 让 3 个 Claude 互相吵架，帮我优化 Claude Code 配置

> 用 Agent Teams 召集一个"AI 评审会"，让三个不同视角的 Claude 互相挑战，帮你做一次配置全面体检。

---

前几天写了一篇 Claude Code 配置指南，发出去之后有人问：我配了 CLAUDE.md，也加了几个 Skill，但总觉得配置"不够好"——有没有什么办法帮我检查一下？

说实话，我自己也有这个感觉。配置这种东西，写的时候觉得挺好，用着用着才发现有些地方不对劲。但你又说不清到底哪里不对。

后来我想到一个办法：**让三个 Claude 互相吵架。**

不是开玩笑。Claude Code 有个实验性功能叫 Agent Teams，可以同时启动多个 Claude 实例，它们各自有独立的上下文，而且 —— 最关键的 —— **它们之间可以直接对话**。

这跟普通的 subagent 不一样。subagent 是你派出去干活的，干完回来汇报给你。Agent Teams 里的 teammate 可以互相发消息，互相挑战、互相反驳。

就像你拉了一个小群，让三个专家在里面讨论你的方案。

## Agent Teams 是什么

一句话：**多个 Claude Code 实例组成的协作团队，它们共享任务列表，互相发消息，独立工作。**

架构长这样：

```
你（Team Lead 会话）
├── Teammate 1（独立的 Claude 实例，有自己的 200K 上下文）
├── Teammate 2（独立的 Claude 实例）
├── Teammate 3（独立的 Claude 实例）
│
├── 通信方式：teammates 之间通过消息互相沟通
└── 协作方式：共享一个任务列表，自行认领任务
```

和 subagent 的区别：

- **Subagent** — 你分别派任务给三个下属，他们各干各的，做完了分别向你汇报。彼此不知道对方在干什么。
- **Agent Teams** — 你召集了一个会议，三个人坐在一起讨论，互相听得到、互相能反驳。最后给你一个共识结论。

什么时候 subagent 更合适？**有明确答案的执行任务**。比如"帮我跑测试"、"review 这段代码"、"生成文档"。一个 agent 就够了，不需要讨论。

什么时候该用 Agent Teams？**没有标准答案、需要多角度碰撞的问题**。比如技术选型、架构决策、配置优化 —— 这些问题一个人想容易有盲区。

## 启用 Agent Teams

设置一个环境变量就行：

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

或者写进 Claude Code 的配置：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

没有额外依赖，没有额外安装。启用后，你只需要用自然语言告诉 Claude "创建一个团队"，它就知道该怎么做了。

## 实操：用 Agent Teams 做配置体检

我拿自己的 Claude Code 配置做了一次完整的体检。以下是我用的 prompt：

```
帮我做一次 Claude Code 配置全面体检。创建一个 Agent Team，spawn 3 个 teammate：

1. 效率专家：审查 CLAUDE.md 指令质量、settings.json 权限设置、model 选择，
   找出拖慢工作流或被 Claude 忽略的配置

2. 安全/规范专家：检查权限是否过松、有没有暴露敏感信息、
   hooks 是否合理、MCP server 配置是否安全

3. 工作流专家：审查 skills 和 commands 是否充分利用、
   有没有缺失的自动化机会、keybindings 有没有可以优化的

要求：
- 每个 teammate 先独立审查，再互相挑战对方的发现
- 如果有分歧，要辩论出结论
- 最终给我一份按优先级排列的改进清单
```

三个 teammate 会各自读取你的配置文件（它们都能访问 CLAUDE.md、settings.json、.claude/ 目录），然后开始工作。

### 辩论过程（精彩的部分）

效率专家先开火："CLAUDE.md 里有 3 条指令互相矛盾 —— 一处说'用中文回复'，另一处说'all output in English'。Claude 每次都在猜该听哪个。"

安全专家跟进："settings.json 的 allowedTools 里有 `Bash(*)`，这等于完全放开了 shell 权限。建议改成白名单。"

工作流专家提了不同意见："如果改成白名单，每次用一个新命令都要手动加 allow，对独立开发者来说太麻烦了。不如加一个 deny 黑名单，只禁止 `rm -rf`、`git push --force` 这种危险操作。"

安全专家反驳："黑名单容易遗漏，你能把所有危险命令都列完吗？"

工作流专家回应："独立开发者的场景下，效率比安全边界更重要。你不是在给一个不信任的实习生配权限，你是在给自己的工具配权限。"

最后三个人达成了一个折中方案：白名单覆盖高频命令，黑名单补充危险操作，其余弹确认框。

**这就是 Agent Teams 的价值 —— 一个人想不到的角度，三个"人"互相一吵就出来了。**

### 最终报告（示例）

Lead 综合三方意见后，给出了一份按优先级排列的改进清单：

**高优先级**
- 修复 CLAUDE.md 中的矛盾指令（效率专家发现）
- 收紧 Bash 权限策略（安全 + 工作流共同建议）

**中优先级**
- 把 3 个重复性操作封装成 Skill（工作流专家建议）
- 将"用中文回复"等通用偏好移到用户级配置（效率专家建议）

**低优先级**
- 添加 SessionStart hook 自动注入项目上下文（工作流专家建议）
- 清理不再使用的 MCP server 配置（安全专家建议）

每一条都标注了是哪个角色提出的，有分歧的地方会注明辩论结论。

## 什么时候该用 Agent Teams

- **技术选型辩论** — 用 PostgreSQL 还是 SQLite？三个 teammate 各支持一个方案，互相挑战
- **疑难 Bug 调试** — 三个 teammate 各提出一个假设，互相尝试推翻对方的理论
- **配置/方案审查** — 就像上面的例子，多角度审视你可能有的盲区

## 什么时候不该用

- **有明确答案的任务** — "帮我写个函数"、"帮我跑测试"，一个 agent 就行，三个人讨论反而浪费
- **需要改同一个文件** — 多个 teammate 同时改同一个文件会冲突
- **简单任务** — 杀鸡焉用牛刀，Agent Teams 的 token 消耗是 teammate 数量的倍数

## 一张图看清三种模式

- **单 Agent** — 你和 Claude 一对一对话。适合日常大部分工作
- **Subagents** — 你派多个 agent 分别执行不同任务，各自汇报给你。适合并行执行明确任务
- **Agent Teams** — 多个 agent 组成团队，互相讨论和协作。适合需要多视角碰撞的开放问题

大部分时候，单 Agent 就够了。需要并行执行的时候用 Subagents。遇到"拿不准"的问题，才需要召集一个 Agent Team 来吵一架。
