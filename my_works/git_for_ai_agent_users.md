# 用 Claude Code 做项目，为什么要学 Git

> Git 是你项目的「时光机 + 平行宇宙」——学会存档和回退，AI 帮你改坏了也不怕。

---

前几天，我让 Claude 帮我重写了一篇文章的开头。

改完我扫了一眼，觉得还不错，就接受了。第二天重新读，越读越别扭——原来的版本虽然粗糙，但有一种真实感；Claude 改完以后，流畅是流畅了，味道却没了。

我想回到原来的版本。

然后我发现，回不去了。

原来的内容已经被覆盖了，Ctrl+Z 也救不了——因为我中间还改过别的文件，编辑器的撤销历史早就断了。

那一刻我的感觉就是：我明明知道之前的版本更好，但它消失了。

后来我才意识到，这个问题不是 Claude 造成的，是我自己没有"存档"的习惯。

## 你可能也遇到过这些场景

我把自己踩过的坑和朋友们聊天时听到的故事整理了一下，看看你中了几条：

**1. CLAUDE.md 改着改着，Claude 的行为变怪了**

你不断调整 CLAUDE.md 里的规则——加一条、删一条、改个措辞。某天你发现 Claude 写出来的东西风格不对了，但你不知道是哪次改动导致的。只能一条条删着试，像在黑暗里摸开关。

**2. 调了 6 版，想回到第 3 版，但已经没了**

你花了一下午让 Claude 帮你调排版模板。v1 不满意，v2 差点意思，v3 挺好但觉得还能更好，接着调到 v6。然后你终于承认：还是第 3 版最好。但第 3 版的内容早就被后面的版本覆盖了。

**3. 发布后数据不好，想对比改了什么**

Claude 帮你重写了公众号文章的开头，你觉得不错就发了。结果数据比平时差了一半，你想对比一下到底改了哪些地方——措辞？结构？还是删了什么关键信息？但你只有"改完之后"的版本，"改之前"的样子你已经想不起来了。

**4. 多人协作，合并改动变灾难**

你和两个同事分别用 Claude Code 维护同一个内容项目。每个人都在本地改文件，到了要汇总的时候——三份文件各有各的改动，有的段落你改了别人也改了，不知道该留哪个。手动对比三份文件，光想想就头疼。

**5. v1/v2/v3... 一周后分不清哪个是哪个**

Claude 每次帮你改完，你习惯性地另存为一个新文件——`article_v1.md`、`article_v2.md`、`article_final.md`、`article_final_final.md`。一周后文件夹里 20 个版本，你打开其中 5 个对比，已经完全分不清每个版本到底改了什么。

**6. 迭代了一个月的写作风格，没留过快照**

你用 Claude 帮你迭代写作风格文件——今天加一条规则，明天调整措辞，后天删掉一段。一个月后你想看看这个风格是怎么一步步演变的、哪些规则是后来加的、哪些被你删过又加回来了——但你从来没存过中间状态。

**7. Claude 偷偷改了你的配置，你几天后才发现**

Claude Code 在执行任务的时候，有时会顺手修改 CLAUDE.md 里的内容。你当时没仔细看就点了确认，过了几天才发现 Claude 的某个行为不对。你想知道"那天它到底改了什么"，但记忆已经模糊了，文件里也只有当前的状态。

---

这些问题的共同点是什么？

**没有"存档"。**

你一直在往前走，但从来没有留下可以回头看的脚印。

## Git 是什么？一句话解释

Git 就是你项目的**「时光机 + 平行宇宙」**。

时光机——你可以随时回到任何一个存档点，看看那个时间点的文件长什么样。

平行宇宙——你可以开一条新的分支去尝试不同方向，试完了不满意，直接扔掉，主线不受影响。

不需要手动复制文件，不需要 `v1/v2/v3` 命名。Git 会帮你记住每一次改动：谁改的、什么时候改的、改了哪些内容。

它是程序员每天都在用的工具。但今天，只要你在用 Claude Code 做项目——哪怕你做的不是代码，是文章、是配置文件、是排版模板——你也需要它。

## 6 个命令，够用了

Git 的功能多到可以写一本书。但对于"用 Claude Code 做项目"这个场景，你只需要学 6 个命令。

### 1. `git init` —— 给项目开启存档功能

在你的项目文件夹里运行一次，Git 就开始工作了：

```bash
cd ~/my-project
git init
```

就这一行。之后这个文件夹里的所有文件变动，Git 都能追踪。

**对应痛点**：场景 6——如果你一开始就 `git init`，后面所有改动都有迹可循。

### 2. `git add` + `git commit` —— 存一个档

Git 不会自动帮你存档，你需要手动告诉它"现在的状态值得记一下"：

```bash
git add my-article.md
git commit -m "完成初稿"
```

`git add` 是选择要存哪些文件的改动，`git commit` 是正式存档，`-m` 后面是这次存档的备注。

每次让 Claude 帮你改东西**之前**，先存一个档。这样改坏了随时能回来。

```bash
# Claude 改之前，存一个档
git add .
git commit -m "Claude 改之前的版本"

# 然后让 Claude 帮你改
# 改完觉得不错，再存一个
git add .
git commit -m "Claude 重写了开头"
```

**对应痛点**：场景 1、2、5——每次改动都有存档，不用靠文件名 `v1/v2/v3` 来区分。

### 3. `git log` —— 看看存了哪些档

```bash
git log --oneline
```

输出大概长这样：

```
a3f8d2c Claude 重写了开头
7b1e4a9 Claude 改之前的版本
e5c2f1d 完成初稿
```

每一行是一个存档点，前面那串字符是它的 ID，后面是你写的备注。

**对应痛点**：场景 6——一个月的迭代历史，一目了然。

### 4. `git diff` —— 看看改了什么

这可能是最实用的命令。它能告诉你两个版本之间到底哪些地方不一样：

```bash
# 看当前改动（还没存档的）
git diff

# 对比两个存档点
git diff 7b1e4a9 a3f8d2c
```

输出会用 `-` 标红删掉的内容，`+` 标绿新增的内容。一眼就能看出 Claude 到底改了你哪些东西。

**对应痛点**：场景 3、7——发布后数据不好，`git diff` 一下就知道 Claude 改了什么。Claude 偷偷改了你的 CLAUDE.md，`git diff` 也能抓到。

### 5. `git checkout` —— 回到之前的版本

想看看某个存档点的内容？

```bash
# 回到某个存档点看看
git checkout 7b1e4a9

# 看完了，回到最新版本
git checkout main
```

想直接把某个文件恢复到之前的版本？

```bash
# 把 article.md 恢复到某个存档点的状态
git checkout 7b1e4a9 -- article.md
```

**对应痛点**：场景 2——想回到第 3 版？找到那个存档点的 ID，一行命令搞定。

### 6. `git branch` —— 开一个平行宇宙

这个命令的用处是：你想试试不同方向，但又不想弄乱现在的版本。

```bash
# 创建一个新分支并切换过去
git checkout -b try-new-style

# 在这里随便折腾……让 Claude 重写、换风格、大改结构
# 改完觉得好，合并回去
git checkout main
git merge try-new-style

# 改完觉得不好，扔掉（-D 大写，强制删除未合并的分支）
git checkout main
git branch -D try-new-style
```

**对应痛点**：场景 4——多人协作时每个人在自己的分支上改，最后用 `git merge` 合并，Git 会自动处理大部分合并工作。

## 让 Claude Code 帮你自动存档

Claude Code 本身不会默认帮你 commit。但你可以在 CLAUDE.md 里加一条规则，让它每次改完文件后自动存档：

```markdown
每次完成文件修改后，自动执行 git add 和 git commit，commit message 简要描述改了什么。
```

加了这条之后，Claude 每次帮你改完东西，都会自动存一个档。你随时可以用 `git log` 看历史，用 `git diff` 对比变化，用 `git checkout` 回退。

前提是——你得先 `git init`。没有 init 的项目，Claude 也没办法帮你存档。

## 入门够了，剩下的用到再学

Git 还有很多功能我没提——`stash`、`rebase`、`cherry-pick`、`tag`……但说实话，我自己日常也就用上面那 6 个命令。

Git 不是程序员的专利。它是所有用 AI 做项目的人的安全网。

你下次打开 Claude Code 之前，先跑一下 `git init` 吧。
