# Claude Code 101：不会写代码，也能用 AI 做事

你正在看的这篇文章，是用 Claude Code 的写作系统生产的。从选题、调研、写初稿到审稿，每一步都是在终端里用自然语言指挥 AI 完成的。整个写作过程没有写一行代码。

如果你的反应是"等等，这怎么做到的？" —— 这就是我想聊的事。

## 聊天框，是 AI 的笼子

我用了很长时间的 ChatGPT 和网页版 Claude。输入一个问题，AI 给一段回答，我再问，它再答。用久了，我一度觉得 AI 就是这样 —— 一个聊天搭子，能写文案、能翻译、能回答问题，但也就这样了。

后来我意识到，问题不在 AI 不够强，而在聊天框把它关住了。

网页版 AI 的活动范围，就是那个对话框。它能接收文字，输出文字，但碰不到电脑上的文件，执行不了任何操作，也连不上其他工具。每次跟它聊，就像去酒店开房 —— 每次都要重新介绍自己，它也不知道你房间长什么样。

Claude Code 不一样。它运行在我的电脑上，能直接读写文件、执行命令、操作项目。Dan Shipper 有个比喻我觉得很准：网页版像住酒店，Claude Code 像有了自己的公寓 —— 它知道东西在哪，能帮你整理，能帮你干活。

说"运行在电脑上"，指的是操作在本地执行 —— 读文件、改文件、跑程序。但 AI 的"思考"仍然在云端完成，所以需要联网。

终端形态给了 AI 更大的施展空间。它可以和其他工具拼接组合，可以处理没有大小限制的文件，可以长时间持续工作 —— 这些是网页聊天框在产品设计上做不到的。而且入口也不只是终端：桌面 app、浏览器都能用。如果对终端有心理障碍，桌面 app 看起来跟普通软件没什么区别。

通过 MCP（Model Context Protocol）这个开放标准，Claude Code 还能连接外部数据源和工具 —— Google Drive、Jira、Slack 等等。它不只是一个操作本地文件的工具，而是一个能伸手到外部世界的 AI 助手。

## 不会写代码，能用来干什么

衡量能不能用 Claude Code 的标准，不是"会不会写代码"，而是"能不能描述你想要什么结果"。

拿这篇文章的写作过程来说。

我搭了一套写作系统，里面有一组预定义的 Skills（可以理解为"工作流程配方"），每个 Skill 用一个斜杠命令触发：

- `/brief`：我描述一个选题，AI 提出几个角度供我选，几轮对话后生成一张任务卡
- `/research`：根据任务卡联网搜索资料，交叉验证事实，生成一份标注了可信度的证据地图
- `/outline`：把任务卡和证据地图组织成文章提纲
- `/draft`：按提纲逐段写出初稿
- `/review`：独立审稿，检查事实准确性、表述强度是否越界

全程中文自然语言交流。我要做的就是在每一步给出判断：这个角度好不好？这个结构对不对？这段表述准不准？AI 负责执行，我负责决策。

这些 Skills 本质上是预先写好的工作流程，一个命令触发一套动作。就像洗衣机 —— 不需要知道内部怎么转，选好模式按开始就行。

搭建这套系统本身需要技术投入，但使用它不需要。这篇文章讲的是"使用"层面 —— 系统搭好之后，操作者只需要会用自然语言描述需求。

写作只是一个例子。已有不少非技术背景的用户在用 Claude Code 做其他事：费用追踪、文件整理、数据分析、批量内容处理。Helen Lee Kupp 的经历很有代表性 —— 她不是开发者，起初对终端充满恐惧，但用了之后发现，"这个工具不是关于编码的"。

这些案例多来自早期采用者，不代表所有人都能立刻上手。基本的电脑操作能力还是要有的，也需要一点耐心适应新的工作方式。

## 怎么找到你的任务

知道 Claude Code 能干什么，跟找到自己用它干什么，中间还隔着一步。这一步需要足够的使用密度来跨过。

有一个叫"30 倍法则"的思维框架：如果每天有一个重复性任务花 10 分钟，花 300 分钟（大约 5 小时）去学习怎么自动化它，30 个工作日后就回本了，剩下的 220 个工作日全是净赚。类比到 Claude Code —— 多花时间摸索它的用法，就是在"培训"你的 AI 助手。这个类比不精确，但方向是对的：前期投入的时间，后面会以倍数还回来。

具体怎么找？我的经验是从三个维度观察：

**重复 3 次法则。** 回顾过去一周，哪些电脑操作重复做了 3 次以上？批量改文件名、从表格里提取数据、把一种格式转成另一种……这些重复劳动都值得交给 Claude Code。

**全天浸泡日。** 找一天不太忙的日子，把所有任务都尝试用 Claude Code 来做。不用期待每件事都成功，目的是发现：哪些出乎意料地顺，哪些完全不行。一天下来就有了一张"适合/不适合"的清单。

**"会说不会做"。** 能用语言清楚描述想要的结果，但不知道具体怎么操作 —— 这类任务是 Claude Code 的甜蜜区。比如"把这 200 张照片按拍摄日期分到不同文件夹"、"从这份 PDF 里提取所有表格数据到 Excel"。我知道要什么，Claude Code 知道怎么做。

这三个方法来自我自己的使用经验，不是什么方法论。但它们指向一个核心：不需要先学会所有东西再开始，先用起来，在使用中发现可能性。

## 为什么要配合 git —— 你的后悔药

前面提到的写作系统里，每一步完成后都会自动"存档"。这个存档系统就是 git。

git 不是程序员专用的神秘工具。把它理解为一个"时间机器式的存档系统"就好：每次存档记录了项目在那个时刻的完整状态，随时可以回到任何一个存档点。

为什么这对 Claude Code 用户特别重要？因为 AI 不是永远正确的。Claude 可能把文件改坏了，可能删了不该删的东西，可能某个方向走偏了。有了 git，这些都不是问题 —— "帮我回到上一步"，一句话搞定。

git 还有一个功能：分支。我可以让 Claude 在一个分支上尝试方案 A，在另一个分支上尝试方案 B，两个方向互不干扰。试完了，我来决定选哪个，把好的合并进来。

不需要先学会 git 命令才能用。Claude Code 会帮你执行 git 操作 —— 用自然语言说"帮我保存当前状态"或者"回到上一步"，它跑对应的命令。学了 git 当然更好，但不是前提条件。

有一点需要提前做：git 本身要装在电脑上。一次性的事，装完就不用管了。

---

我正在看的这篇文章，就是一个活着的例子。

从 AI 拿到的，很可能只是它能给的一小部分。扩大这个空间的门槛，比想象的低得多。

不需要先学会终端，不需要先学会 git，不需要先学会任何你觉得"应该先学会"的东西。找一个任务，用自然语言告诉 Claude 你想要什么，然后看看会发生什么。
